WE DO · 1
Build the image, stamp it with its commit, prove the cache
Build the gateway image with a real provenance stamp, then prove the layer cache behaves like the teaching slide said.
1
scripts/build-image.sh builds and tags :sha-<short> + :local. It passes --build-arg GIT_SHA=$(git rev-parse --short HEAD) for you.
2
docker history shows the baked layers, newest on top. Which layer has the biggest size?
3
Read the revision label back and confirm it matches the last commit. That's how you trace any container to its commit.
4
Change a project file (e.g. change the perspective view size) and rebuild. Docker checksums the content of each COPY's inputs: the changed bytes bust the projects/ layer, but every layer above it stays CACHED, so the rebuild takes seconds.
5
Now drop a dummy file into third-party-modules/ and rebuild. That COPY sits higher in the Dockerfile, and a busted layer rebuilds everything after it: modules, config and projects all re-copy. Delete the dummy afterwards — or it ships in every future build.
scripts/build-image.sh
docker history cicd-lab-05-ignition:local

# step 3: trace image to its commit
docker inspect -f '{{ index .Config.Labels
  "org.opencontainers.image.revision" }}' \
  cicd-lab-05-ignition:local
git rev-parse --short HEAD

# step 4: a content change busts projects/ only
scripts/build-image.sh

# step 5: a new file higher up busts much more
echo test > third-party-modules/dummy.txt
scripts/build-image.sh
rm third-party-modules/dummy.txt
Layer order

A busted layer busts everything after it. Least-volatile first, most-volatile last: that's why your daily project edits rebuild in seconds. And "changed" means bytes, not metadata — the cache checksums content, so a touch busts nothing.

Run the image with nothing mounted, and read what got baked
The self-contained proof: the project shows up because it's inside the image, with no ./projects mount and no scan call.
1
Run the image on a spare port with no bind mounts (see command). A gateway boot takes ~30 to 60 s — start it now, let it run, and carry on with the next steps while it comes up.
2
Once the log prints "Starting project: example-project", open http://localhost:9088 → example-project is there. Ctrl-C stops it; --rm cleans up.
3
Meanwhile: see what .dockerignore kept out. Your build output already measured it: the transferring context: ~60 MB line, versus du -sh . ≈ 130 MB on disk. The gap — mostly .git — never reached Docker.
4
Explore what actually got baked: read the image, not the repo. Open the image in Docker Desktop (Images → cicd-lab-05-ignition → Files) — or whatever image browser you prefer — and find all four COPYed artifacts: data/projects/, data/config/, data/modules.json, /third-party-modules. Note what's not there: README.md, docs/, .env.
# step 1: the self-contained proof
docker run --rm --user root -p 9088:8088 \
  -e ACCEPT_IGNITION_EULA=Y -e IGNITION_EDITION=standard \
  -e GATEWAY_ADMIN_USERNAME=admin \
  -e GATEWAY_ADMIN_PASSWORD=password \
  cicd-lab-05-ignition:local \
  -n demo -- -Dignition.allowunsignedmodules=true

# step 3: check .git folder size
du -sh .git

Part 1.1
PART 1.1
Bake a one-line change into the image
Change the artifact, not a running gateway — then prove the change is in the image without ever starting one.
1
Open projects/example-project/project.json — the project's metadata: title, description, enabled.
2
Change the description to something unmistakably yours, e.g. "Example Project — baked by <you>".
3
Rebuild: scripts/build-image.sh. Watch the output: only the projects/ layer rebuilds — the we-do cache lesson, live.
4
Prove it's baked in without running a gateway: override the entrypoint so the container runs cat instead of booting.
# step 4: read the file out of the image layer
docker run --rm --entrypoint cat \
  cicd-lab-05-ignition:local \
  /usr/local/bin/ignition/data/projects/example-project/project.json
The point

The change is now part of the artifact. Every gateway that ever runs this image boots with it: no copy step to script, no file to forget. That's what "the image is the deploy spec" means.

Part 1.2
PART 1.2
Prove a secret leaks into the build context, then fence it out
The context is everything Docker can see. One careless COPY . . — now, or in a refactor two years from now — and anything in it ships inside a pullable artifact. .dockerignore is the backstop.
1
Play the villain: echo secret > NOTES.local — a stand-in for the scratch notes, exports, and credential files that accumulate in every real repo.
2
See it leak. Dump the context with the probe build from the we-do: NOTES.local is in there. Nothing bakes it yet — the Dockerfile COPYs four fixed paths — but it's one careless COPY away from a published image.
3
Fence it out. Add a pattern (e.g. *.local) to .dockerignore, re-run the probe → gone.
4
Re-run scripts/validate.sh: must stay green (your pattern didn't remove the required exclusions). Delete NOTES.local when done.
echo secret > NOTES.local

# the probe: what can Docker see?
docker build -q -t ctx-probe -f- . <<'EOF'
FROM busybox
COPY . /ctx
EOF
docker run --rm ctx-probe ls /ctx   # NOTES.local — leaked

# add "*.local" to .dockerignore → re-probe: gone
scripts/validate.sh
rm NOTES.local
Why this matters

The line in the shipped .dockerignore that really matters is .env: an image that bakes credentials publishes them the moment it's pushed to a registry. .dockerignore isn't a speed tweak — it's the backstop that makes that accident impossible.

Part 1.3
PART 1.3
Give one build several tags, and trace it to its commit
A tag is just a name pointing at an image ID. Pipelines lean on two kinds: moving names like :dev, and immutable names like :sha-… or a release version like :v1.4.2.
1
Build with an extra moving tag, the way a pipeline would tag :dev.
2
docker images → three tags, one image ID: :sha-<short>, :local, :dev. Same bytes, three names.
3
Read the revision label off the :sha-<short> tag; confirm it matches git rev-parse --short HEAD.
# moving + immutable tags coexist
scripts/build-image.sh --tag dev
docker images cicd-lab-05-ignition
# :sha-<short>  :local  :dev all share one IMAGE ID

# trace the immutable tag to its commit
docker inspect -f '{{ index .Config.Labels
  "org.opencontainers.image.revision" }}' \
  cicd-lab-05-ignition:sha-<short>
git rev-parse --short HEAD
Foreshadow

Immutable tags are what make part 2's rollback trivial: every build keeps a name that never moves, so "go back" means "run yesterday's name". :sha-… is the machine's immutable name; a semver tag like :v1.4.2 is the human's — this repo's release.yml mints one per vX.Y.Z git tag.

2 · READ FIRST
Why we deploy locally today, and what GitHub Actions would do instead
In a real setup this part is automated by the same Actions you built in Lab 04. Today you type the workflow's commands yourself.
  • Same triggers as Lab 04. A merge to main would fire deploy.yml, a tag would fire release.yml. Only the steps inside the jobs change: build + push + pull + recreate, instead of copy + scan.
  • Why not do that now? The gateway image is ~2 GB. Between GitHub's datacenter and a registry that upload is seconds; from ten course laptops, it's your whole hour. The transport isn't the lesson.
  • So today, you are the pipeline. Every command in Parts 2.1 to 2.3 is a line the workflow would run. Type them once by hand and the automation never feels like magic again.
# what deploy.yml's deploy job would run
# (on the runner that owns the gateway):
docker pull ghcr.io/<you>/<repo>:sha-<short>
IGNITION_DEV_IMAGE=ghcr.io/<you>/<repo>:sha-<short> \
  docker compose up -d ignition-dev

# what YOU run today (no registry, so no pull):
IGNITION_DEV_IMAGE=cicd-lab-05-ignition:local \
  docker compose up -d ignition-dev
Same deploy

The only line missing locally is docker pull: your image store already holds the build. Everything else is identical to what the workflow runs. Want to run the real thing? The stretch slide at the end walks the full CI path.

Part 2.1
PART 2.1
Deploy your image to the dev gateway, command by command
No script today: you type the deploy yourself so you see there's nothing to it. No docker cp, no scan call: the container is replaced.
1
Confirm dev is on the base image (an empty gateway).
2
The deploy is one command: point the compose variable at your image and bring the service up. Compose sees a new image and recreates the container.
3
Wait for the gateway to come back: poll /StatusPing until it says RUNNING (a fresh-image boot re-commissions, so give it a few minutes).
4
Verify at http://localhost:8089: example-project is there — and so is your part 1 description (docker exec lab05-ignition-dev cat data/projects/example-project/project.json). Then inspect: which image does dev run now?
5
Note what persisted: the historian data in TimescaleDB is untouched. The container died; the data that matters didn't live in it.
# 1: before, the base image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev

# 2: THE deploy
IGNITION_DEV_IMAGE=cicd-lab-05-ignition:local \
  docker compose up -d ignition-dev

# 3: wait for RUNNING
curl -s localhost:8089/StatusPing
# repeat until it prints RUNNING

# 4: after, your image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev

# and your baked part 1 description came along
docker exec lab05-ignition-dev \
  cat data/projects/example-project/project.json
Under the hood

docker-compose.yaml interpolates IGNITION_DEV_IMAGE into the dev service's image:. New image → compose recreates the container. That's the whole deploy; in production the only extra ingredient is a docker pull first.

Part 2.2
PART 2.2
Ship a change end-to-end: edit, commit, rebuild, redeploy
The full loop, exactly what a pipeline would automate. Commit before you build, so each build gets its own immutable :sha tag: that's your rollback point in Part 2.3.
1
Note the current build's tag: docker images cicd-lab-05-ignition. Write down the :sha-<short>: this is the version you'll roll back to.
2
Edit a Perspective view under projects/example-project/…/views/: make it visibly different (a title text works well).
3
Commit the change on a feature branch and merge it to main (GitHub flow, like every lab). The commit is what gives the next build a new :sha tag.
4
Rebuild, then redeploy with the same one-liner from Part 2.1: scripts/build-image.sh, then compose up with the new build.
5
Verify at :8089: your view change is live. docker images now shows two :sha-… tags: two deployable versions, both still on your machine.
# 1: your rollback point
docker images cicd-lab-05-ignition
# note the current :sha-<short> tag

# 3: commit so the next build gets a new sha
git checkout -b feature/tweak-view
# edit the view.json, then:
git add -A && git commit -m "Change overview title"
git checkout main && git merge feature/tweak-view

# 4: rebuild + redeploy
scripts/build-image.sh
IGNITION_DEV_IMAGE=cicd-lab-05-ignition:local \
  docker compose up -d ignition-dev
This is the pipeline

Steps 2-5 are exactly what a deploy workflow automates on merge: build → tag → run. You're doing the pipeline's job by hand once, so the automation never feels like magic.

Part 2.3
PART 2.3
Roll back by running the previous build's tag
The image-based rollback: point the container at yesterday's name. No git surgery, no re-copying files, nothing rebuilt.
1
Pretend Part 2.2's change broke something on dev. You need the previous version back, now.
2
Deploy the old :sha-<short> tag you wrote down in Part 2.2: the same one-liner, older name. The old image never went away; it's a name in your image store.
3
Verify at :8089: the view is back to its previous state.
4
Count the irreversible steps you just took. (It's zero: both versions still exist, and you can flip between them all afternoon.)
# 2: rollback = the same deploy, older tag
IGNITION_DEV_IMAGE=cicd-lab-05-ignition:sha-<old-sha> \
  docker compose up -d ignition-dev

# 3: confirm what dev runs now
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev
Compare

File-based rollback (Lab 04) meant re-running the pipeline on an older tag: it works, but it re-copies and re-scans. Here the previous version already exists as an artifact: rollback is just running it. That difference is the whole reason teams keep versioned images around.

Stretch · optional
STRETCH
Run the real pipeline through GitHub Actions
Everything you typed in part 2 is automated in this repo's workflows — GitHub flow: merge to main → dev, tag → prod. Needs your fork + the bundled runner; also a great take-home if the venue Wi-Fi hates 2 GB pulls.
1
One-time setup: in .env, point RUNNER_REPO_URL at your fork and paste your Lab 04 repo-scope PAT into RUNNER_GITHUB_PAT, then start the runner. It appears under your fork's Settings → Actions → Runners.
2
Ship to dev via CI: open a PR with a small project change and watch ci.yml validate it (including a no-push image build). Merge to main → deploy.yml builds on a free hosted runner, pushes to your GHCR, and your runner recreates dev (:8089).
3
Release to prod: push a v* tag → release.yml re-tags :dev to :v0.1.0 + :prod — no rebuild — and recreates prod (:8090).
4
The payoff: prove dev and prod run the same digest. Then roll prod back with one click: Actions → Release → Run workflow, entering an earlier version.
# 1: register the bundled runner against YOUR fork
#    .env → RUNNER_REPO_URL + RUNNER_GITHUB_PAT
docker compose up -d github-runner

# 2: ship — PR → merge to main → watch Actions

# 3: release
git tag v0.1.0 && git push origin v0.1.0

# 4: the digest-equality proof
docker inspect -f '{{ index .RepoDigests 0 }}' \
  lab05-ignition-dev
docker inspect -f '{{ index .RepoDigests 0 }}' \
  lab05-ignition-prod
# identical sha256 → prod runs the bytes dev tested
Build-once, promote-many

The release never rebuilds: it re-tags the image dev validated, so prod runs those exact bytes. That digest-equality check is the whole philosophy of image-based deploy in one command pair.
