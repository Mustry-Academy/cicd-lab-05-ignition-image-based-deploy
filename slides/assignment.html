WE DO · 1
Build it, stamp it, cache it
Build the gateway image with a real provenance stamp, then prove the layer cache behaves like the slide said.
1
scripts/build-image.sh — builds and tags :sha-<short> + :local. It passes --build-arg GIT_SHA=$(git rev-parse --short HEAD) for you.
2
docker history — see the baked layers, newest on top. Which one is biggest?
3
Read the revision label back and confirm it matches git rev-parse --short HEAD — that's how you trace any container to its commit.
4
touch a project file, rebuild → the modules/config layers say CACHED; only projects/ rebuilds.
5
Now touch something in third-party-modules/ and rebuild — notice more layers bust. Why?
scripts/build-image.sh
docker history cicd-lab-05-ignition:local

# step 3 — trace image → commit
docker inspect -f '{{ index .Config.Labels
  "org.opencontainers.image.revision" }}' \
  cicd-lab-05-ignition:local

# step 4 — bust only the bottom layer
touch projects/example-project/project.json
scripts/build-image.sh   # watch for CACHED
Layer order

A busted layer busts everything after it. Least-volatile first, most-volatile last — that's why your daily project edits rebuild in seconds.

WE DO · 2
Read the image, then run it with nothing mounted
The self-contained proof: the project shows up because it's inside the image — no ./projects mount, no scan call.
1
See what .dockerignore kept out of the context: du -sh .git is usually the biggest single chunk.
2
List what actually got baked — read the image, not the repo. Note the --entrypoint: the IA image otherwise treats trailing args as gateway arguments.
3
Run the image on a spare port with no bind mounts (see command). Wait ~30–60 s for "Starting project: example-project" in the log.
4
Open http://localhost:9088 → example-project is there. Ctrl-C stops it; --rm cleans up.
# step 2 — what's inside the image?
docker run --rm --entrypoint find \
  cicd-lab-05-ignition:local \
  /usr/local/bin/ignition/data/projects -maxdepth 2

# step 3 — the self-contained proof
docker run --rm --user root -p 9088:8088 \
  -e ACCEPT_IGNITION_EULA=Y -e IGNITION_EDITION=standard \
  -e GATEWAY_ADMIN_USERNAME=admin \
  -e GATEWAY_ADMIN_PASSWORD=lab05password \
  cicd-lab-05-ignition:local \
  -n demo -- -Dignition.allowunsignedmodules=true
--user root

Drop it and the gateway FAULTs (unable to create resource dir … /.resources): the baked files are root-owned. File ownership is part of what you bake into an image.

Block C · Part 1
C · PART 1
Bake in a module change
Enable a module through the image — the change a project scan can never apply.
1
Open services/modules.json and the compose GATEWAY_MODULES_ENABLED list — see what's available vs. what's on.
2
Enable one more module (edit services/modules.json, or confirm it's in the enabled list).
3
Rebuild: scripts/build-image.sh.
4
Prove it's baked in without running a gateway — override the entrypoint so the container runs cat instead of booting.
# step 4 — read the manifest out of the image layer
docker run --rm --entrypoint cat \
  cicd-lab-05-ignition:local \
  /usr/local/bin/ignition/data/modules.json
The point

In Lab 04, getting this onto a running gateway needed a restart, not a scan. Here the restart is the deploy — the change is a layer in the image.

C · PARTS 2–3
Guard the context, tag like CI
Part 2: keep a secret out of the build context. Part 3: give one image several names, like deploy.yml does.
1
Part 2. Add a throwaway file that must never reach the image: echo secret > NOTES.local.
2
Reason about it: no current pattern matches NOTES.local — so it would be in the context. Add a pattern (e.g. *.local) to .dockerignore.
3
Re-run scripts/validate.sh — the sanity check must still pass (your pattern didn't remove required ones). Delete NOTES.local when done.
4
Part 3. Build with an extra moving tag, the way CI tags :dev — three tags, one image ID.
5
Read the revision label off the :sha-<short> tag; confirm it matches git rev-parse --short HEAD.
# part 2 — the context guard
echo secret > NOTES.local
# add "*.local" to .dockerignore, then:
scripts/validate.sh
rm NOTES.local

# part 3 — moving + immutable tags coexist
scripts/build-image.sh --tag dev
docker images cicd-lab-05-ignition
# :sha-<short>  :local  :dev → same IMAGE ID
Foreshadow

Moving tags (:dev) and immutable tags (:sha-…) coexisting is the core of Block D's deploy/promote story.

We do · together · Block D
WE DO · D
Deploy to dev, by hand
Do manually what deploy.yml automates: recreate the dev gateway from your local image. Purely local — no registry involved.
1
Confirm dev is on the base image (an empty gateway).
2
Recreate dev from your local build — watch it recreate the container and wait for RUNNING.
3
Verify at http://localhost:8089 — example-project is there. No docker cp, no scan. The container was replaced.
4
Inspect again — which image does dev run now?
5
Note what persisted: the historian data in TimescaleDB is untouched. The container died; the data that matters didn't live in it.
# 1 — before: base image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev

# 2 — the deploy (pull → recreate → wait)
scripts/deploy-image.sh dev cicd-lab-05-ignition:local

# 4 — after: your image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev
Under the hood

The script sets IGNITION_DEV_IMAGE and runs docker compose up -d ignition-dev — compose sees a new image and recreates the container. That's the whole deploy.

Runner online, environments created
The plumbing check before anything can deploy.
1
Part 1. Check the bundled runner (see commands) — look for "Listening for Jobs".

In your fork: Settings → Actions → Runners should show it online with self-hosted, lab05. If not: RUNNER_REPO_URL must point at your fork, PAT must be real, then docker compose restart github-runner.

# part 1 — is the runner alive?

# → "Listening for Jobs"
Why environments

Even empty they earn their keep: per-stage deploy history, and the place to add required reviewers on prod later (stretch). No IGNITION_API_KEY here — that was a Lab 04 thing; image-based replaces the container instead of scanning it.

Block D · Part 3
D · PART 3
Ship to dev via develop
The Git Flow loop: feature branch → PR into develop → merge fires deploy.yml → dev gateway is recreated.
1
One-time: your fork needs a develop branch (see command). Optionally make it the default branch so PRs target it.
2
Branch feature/tweak-view off develop, edit a Perspective view under projects/example-project/…/views/, commit, push.
3
Open a PR into develop — watch ci.yml: JSON, hadolint, actionlint, and a no-push image build.
4
Merge → deploy.yml fires. build (hosted) pushes :sha-<short> + :dev to your GHCR; deploy (self-hosted) pulls and recreates dev.
5
Verify at :8089 — your view change is live. Then confirm which image dev runs.
# 1 — one-time Git Flow setup
git checkout -b develop && git push -u origin develop

# 2 — the feature loop
git checkout develop && git checkout -b feature/tweak-view
# edit a view.json → commit → push → PR into develop

# 5 — after the deploy
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev
# → ghcr.io/<you>/cicd-lab-05-ignition:sha-<short>
Nothing ran?

Actions are disabled on forks by default. Actions tab → enable. Then check your fork → Packages: the image lands under your namespace, never the instructor's. 403? → docs/TROUBLESHOOTING.md.

Block D · Part 4
D · PART 4
Release to prod — and prove the digest
Cut a release the Git Flow way: merge develop → main, tag it, and watch prod get the image dev tested — not a rebuild.
1
Merge and tag (see commands). A plain merge stands in for a release/* branch here.
2
release.yml fires on the tag: promote re-tags :dev → :v0.1.0 + :prod (no rebuild!); deploy recreates prod from :v0.1.0.
3
Note: it promotes :dev, not the main commit — the merge commit has no image of its own.
4
The money moment: compare digests on dev and prod. The sha256:… must match — tags differ, content doesn't.
# 1 — merge develop into main, then tag
git checkout main && git pull
git merge --no-ff develop -m "Release v0.1.0"
git push origin main
git tag v0.1.0 && git push origin v0.1.0

# 4 — prove build-once / promote-many
docker inspect -f '{{ index .RepoDigests 0 }}' \
  lab05-ignition-dev
docker inspect -f '{{ index .RepoDigests 0 }}' \
  lab05-ignition-prod
# → identical sha256 digests ✅
Ask yourself

What would it take for these digests to differ — and why would that scare you? (Hint: a rebuild could pull a newer base layer dev never tested.)

Block D · Part 5
D · PART 5
Roll back — without touching git
The image-based rollback button: run the previous tag. Two clicks, zero irreversible steps.
1
First, make something to roll back from: ship another change through develop → dev, merge to main, then git tag v0.2.0 && git push origin v0.2.0. Prod now runs v0.2.0.
2
In your fork: Actions → Release → Run workflow, set the version input to v0.1.0.
3
promote re-tags v0.1.0's existing image back to :prod; deploy recreates prod from it. No rebuild, no git revert, no history surgery.
4
Verify prod reverted — the view is back to its v0.1.0 state.
# 1 — second release so there's something to undo
# (change → develop → merge to main, then:)
git tag v0.2.0 && git push origin v0.2.0

# 2–3 — rollback = re-promote an old tag
# GitHub UI: Actions → Release → Run workflow
#            tag: v0.1.0

# 4 — confirm
docker inspect -f '{{.Config.Image}}' lab05-ignition-prod
Compare

In Lab 04, rolling prod back meant re-copying old files and re-scanning. Count the irreversible steps in each path — image-based has zero: you point at an artifact that already exists.

Block D definition of done
Walk the list in exercises/block-d.md — the digest proof is the one we'll ask you about.

Stretch — Block C
Finished early? Three ways to go deeper on the image itself.
1
Shrink the context. Build with --progress=plain and watch the "transferring context" size. Add more to .dockerignore and watch it drop.
2
Layer forensics. docker history --no-trunc — which layer is largest: base, modules, or your project? What would you do differently if the project were 500 MB of assets?
3
Multi-stage thinking. Our Dockerfile is single-stage because the IA base image is the runtime. When would you want a builder stage? (Hint: compiling a custom module from source, or tooling you don't want in the shipped image.)
docker build --progress=plain . 2>&1 | \
  grep "transferring context"

docker history --no-trunc \
  cicd-lab-05-ignition:local
No single answer

The 500 MB question has no right answer — bake vs mount vs pull-at-runtime is the trade-off itself. Bring your reasoning to the debrief.

Stretch — Block D
The production-shaped extras.
1
Approval gate. Add a required reviewer to lab-gateway-prod (Settings → Environments → Required reviewers). Push a tag and watch the deploy job wait for approval — no workflow change needed. That's the point of environments.
2
Rollback math. You rolled back in two clicks. Write out the Lab 04 file-based rollback sequence step by step and compare the failure surface of each.
3
Make the package public. Flip the GHCR package to public, then docker pull from a machine that's never logged in. When would you not want that?
Debrief questions

Bring answers to these: · why deploy the immutable :sha-… and not :dev? · which failure modes does replace-the-container remove versus rm -rf + docker cp? · where do gateway audit logs / user accounts live in the image-vs-volume-vs-database model? · the runner has the Docker socket — how would "pull + recreate" reach a gateway you can't give socket access to?
