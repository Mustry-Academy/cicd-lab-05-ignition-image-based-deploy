The assignment · breakout rooms · 2 × 40 min
Build the gateway image locally, then deploy it by replacing the container
Two blocks, everything on your own machine. In Block C you build the gateway image, bake a change into it, and prove it's self-contained: no bind mounts, no scan call. In Block D you deploy that image to the local dev gateway by recreating the container, ship a second change end-to-end, and roll back by running the previous build's tag. No registry, no uploads, no GitHub setup needed.

How this works
  • Share your screen in your breakout room, and we'll drop in where it's useful.
  • The exercise files are the source of truth: exercises/block-c.md and exercises/block-d.md. These slides are the map, not the territory.
  • Pre-flight: cp .env.example .env then scripts/setup.sh (idempotent, so it's safe if the stack is already up). You need Docker with Compose v2 and ≥ 8 GB free RAM.
  • Everything runs locally today. No fork with Actions, no PAT, no registry account: a gateway image is ~2 GB, and uploading that from a course laptop is not a good use of your hour. The registry legs of the pipeline were covered in the teaching; here you do build, tag and run.
  • Stuck? docs/TROUBLESHOOTING.md first; most issues are one of the first three entries. Don't read instructor-notes/ until you've finished the solo work.
  • Trial mode: gateways run 2-hour trials; reset via Config → Licensing → Reset Trial. The dev gateway gets a fresh trial clock on every deploy anyway (fresh container).
Ask early

Stuck for more than a couple of minutes? Say so in the room. Don't sit blocked in silence; that's exactly what we're here for.

Build the image, stamp it with its commit, prove the cache
Build the gateway image with a real provenance stamp, then prove the layer cache behaves like the teaching slide said.
1
scripts/build-image.sh builds and tags :sha-<short> + :local. It passes --build-arg GIT_SHA=$(git rev-parse --short HEAD) for you.
2
docker history shows the baked layers, newest on top. Which one is biggest?
3
Read the revision label back and confirm it matches git rev-parse --short HEAD. That's how you trace any container to its commit.
4
touch a project file, rebuild → the modules/config layers say CACHED; only projects/ rebuilds.
5
Now touch something in third-party-modules/ and rebuild, and notice more layers bust. Why?
scripts/build-image.sh
docker history cicd-lab-05-ignition:local

# step 3: trace image to its commit
docker inspect -f '{{ index .Config.Labels
  "org.opencontainers.image.revision" }}' \
  cicd-lab-05-ignition:local

# step 4: bust only the bottom layer
touch projects/example-project/project.json
scripts/build-image.sh   # watch for CACHED
Layer order

A busted layer busts everything after it. Least-volatile first, most-volatile last: that's why your daily project edits rebuild in seconds.

WE DO · 2
Read what got baked, then run the image with nothing mounted
The self-contained proof: the project shows up because it's inside the image, with no ./projects mount and no scan call.
1
See what .dockerignore kept out of the context: du -sh .git is usually the biggest single chunk.
2
List what actually got baked: read the image, not the repo. Note the --entrypoint: the IA image otherwise treats trailing args as gateway arguments.
3
Run the image on a spare port with no bind mounts (see command). Wait ~30 to 60 s for "Starting project: example-project" in the log.
4
Open http://localhost:9088 → example-project is there. Ctrl-C stops it; --rm cleans up.
# step 2: what's inside the image?
docker run --rm --entrypoint find \
  cicd-lab-05-ignition:local \
  /usr/local/bin/ignition/data/projects -maxdepth 2

# step 3: the self-contained proof
docker run --rm --user root -p 9088:8088 \
  -e ACCEPT_IGNITION_EULA=Y -e IGNITION_EDITION=standard \
  -e GATEWAY_ADMIN_USERNAME=admin \
  -e GATEWAY_ADMIN_PASSWORD=lab05password \
  cicd-lab-05-ignition:local \
  -n demo -- -Dignition.allowunsignedmodules=true
--user root

Drop it and the gateway FAULTs (unable to create resource dir … /.resources): the baked files are root-owned. File ownership is part of what you bake into an image: remember the COPY --chown lines in the Wilms Dockerfile.

Block C · solo · 40 min · exercises/block-c.md
Bake a change into the image and prove it's self-contained

Four parts, in order. Everything runs locally, no GitHub needed.

  • Part 1: bake a module-manifest change into the image and read it back (15 min).
  • Part 2: extend .dockerignore and keep validate.sh green (10 min).
  • Part 3: give one build several tags, and trace it to its commit (10 min).
  • Part 4: break the build on purpose (5 min). Failures move left, to build time.
Mustry Academy · CI/CD for IgnitionBlock C · Parts 1 → 4
Block C · Part 1
C · PART 1
Bake a module-manifest change into the image
Change what the gateway boots with by changing the artifact, then prove the change is in the image without ever starting a gateway.
1
Open services/modules.json and the compose GATEWAY_MODULES_ENABLED list, and see what's available vs. what's on.
2
Enable one more module (edit services/modules.json, or confirm it's in the enabled list).
3
Rebuild: scripts/build-image.sh.
4
Prove it's baked in without running a gateway: override the entrypoint so the container runs cat instead of booting.
# step 4: read the manifest out of the image layer
docker run --rm --entrypoint cat \
  cicd-lab-05-ignition:local \
  /usr/local/bin/ignition/data/modules.json
The point

The change is now part of the artifact. Every gateway that ever runs this image boots with it: no copy step to script, no file to forget. That's what "the image is the deploy spec" means.

Block C · Parts 2-3
C · PARTS 2-3
Extend .dockerignore, then give one build several tags
Part 2: keep a file that must never ship out of the build context. Part 3: see that tags are just names, and one of them should always be immutable.
1
Part 2. Add a throwaway file that must never reach the image: echo secret > NOTES.local.
2
Reason about it: no current pattern matches NOTES.local, so it would be in the context. Add a pattern (e.g. *.local) to .dockerignore.
3
Re-run scripts/validate.sh: the sanity check must still pass (your pattern didn't remove required ones). Delete NOTES.local when done.
4
Part 3. Build with an extra moving tag: three tags, one image ID.
5
Read the revision label off the :sha-<short> tag; confirm it matches git rev-parse --short HEAD.
# part 2: the context guard
echo secret > NOTES.local
# add "*.local" to .dockerignore, then:
scripts/validate.sh
rm NOTES.local

# part 3: moving + immutable tags coexist
scripts/build-image.sh --tag dev
docker images cicd-lab-05-ignition
# :sha-<short>  :local  :dev all share one IMAGE ID
Foreshadow

The immutable :sha-… tag is what makes Block D's rollback trivial: every build keeps a name that never moves, so "go back" means "run yesterday's name".

Block D · solo · 40 min · exercises/block-d.md
Deploy your image to the dev gateway by replacing the container

The deploy you learned in the teaching, done for real on your machine: recreate the container from a new image, verify, then roll back. No registry involved: your local image store plays that role today.

  • Part 1: deploy your local image to the dev gateway (10 min).
  • Part 2: ship a change end-to-end: edit → commit → rebuild → redeploy (15 min).
  • Part 3: roll back to the previous build's tag, and check what survived (10 min).
Mustry Academy · CI/CD for IgnitionBlock D · Parts 1 → 3
Block D · Part 1
D · PART 1
Deploy your image to the dev gateway
Recreate the dev container from your local build. No docker cp, no scan call: the container is replaced.
1
Confirm dev is on the base image (an empty gateway).
2
Recreate dev from your local build, watching it recreate the container and waiting for RUNNING.
3
Verify at http://localhost:8089: example-project is there, plus the manifest change you baked in Block C.
4
Inspect again: which image does dev run now?
5
Note what persisted: the historian data in TimescaleDB is untouched. The container died; the data that matters didn't live in it.
# 1: before, base image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev

# 2: the deploy (recreate, wait)
scripts/deploy-image.sh dev cicd-lab-05-ignition:local

# 4: after, your image
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev
Under the hood

The script sets IGNITION_DEV_IMAGE and runs docker compose up -d ignition-dev: compose sees a new image and recreates the container. That's the whole deploy. In production the only extra ingredient is a registry between build and run.

Block D · Part 2
D · PART 2
Ship a change end-to-end: edit, commit, rebuild, redeploy
The full loop, exactly what a pipeline would automate. Commit before you build, so each build gets its own immutable :sha tag: that's your rollback point in Part 3.
1
Note the current build's tag: docker images cicd-lab-05-ignition. Write down the :sha-<short>: this is the version you'll roll back to.
2
Edit a Perspective view under projects/example-project/…/views/: make it visibly different (a title text works well).
3
Commit the change on a feature branch and merge it to main (GitHub flow, like every lab). The commit is what gives the next build a new :sha tag.
4
Rebuild + redeploy: scripts/build-image.sh, then scripts/deploy-image.sh dev cicd-lab-05-ignition:local.
5
Verify at :8089: your view change is live. docker images now shows two :sha-… tags: two deployable versions, both still on your machine.
# 1: your rollback point
docker images cicd-lab-05-ignition
# note the current :sha-<short> tag

# 3: commit so the next build gets a new sha
git checkout -b feature/tweak-view
# edit the view.json, then:
git add -A && git commit -m "Change overview title"
git checkout main && git merge feature/tweak-view

# 4: rebuild + redeploy
scripts/build-image.sh
scripts/deploy-image.sh dev cicd-lab-05-ignition:local
This is the pipeline

Steps 2-5 are exactly what a deploy workflow automates on merge: build → tag → run. You're doing the pipeline's job by hand once, so the automation never feels like magic.

Mustry Academy · CI/CD for IgnitionPart 2 of 3 · 15 min
Block D · Part 3
D · PART 3
Roll back by running the previous build's tag
The image-based rollback: point the container at yesterday's name. No git surgery, no re-copying files, nothing rebuilt.
1
Pretend Part 2's change broke something on dev. You need the previous version back, now.
2
Deploy the old :sha-<short> tag you wrote down in Part 2 (see command). The old image never went away; it's a name in your image store.
3
Verify at :8089: the view is back to its previous state.
4
Count the irreversible steps you just took. (It's zero: both versions still exist, and you can flip between them all afternoon.)
# 2: rollback = deploy an older immutable tag
scripts/deploy-image.sh dev \
  cicd-lab-05-ignition:sha-<the-old-short-sha>

# 3: confirm what dev runs now
docker inspect -f '{{.Config.Image}}' lab05-ignition-dev
Compare

File-based rollback (Lab 04) meant re-running the pipeline on an older tag: it works, but it re-copies and re-scans. Here the previous version already exists as an artifact: rollback is just running it. That difference is the whole reason teams keep versioned images around.

Fresh gateway alert

Each deploy gave dev a brand-new gateway: new internal DB, new trial clock, gone are any UI-made edits. If that surprised you, good: that's the "fresh gateway" disadvantage from the teaching, experienced first-hand.

Stretch: go deeper on the image itself
Finished early? Three ways to sharpen the Block C skills.
1
Shrink the context. Build with --progress=plain and watch the "transferring context" size. Add more to .dockerignore and watch it drop.
2
Layer forensics. Run docker history --no-trunc: which layer is largest, base, modules, or your project? What would you do differently if the project were 500 MB of assets?
3
CI build check. If you have a fork with Actions enabled: add a job to ci.yml that runs docker build without pushing. A broken Dockerfile now fails the PR: free, because nothing 2 GB ever leaves the runner.
docker build --progress=plain . 2>&1 | \
  grep "transferring context"

docker history --no-trunc \
  cicd-lab-05-ignition:local
No single answer

The 500 MB question has no right answer: bake vs mount vs pull-at-runtime is the trade-off itself. Bring your reasoning to the debrief.
