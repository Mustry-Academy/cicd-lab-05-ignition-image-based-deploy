Deploying Ignition as a custom Docker image
Previous week you shipped files into a running gateway. This lab is a deliberate sidestep: there is a second way to deploy the exact same files, by baking them into a custom Docker image and replacing the container. To understand it you first need three words defined properly: image, Dockerfile and container.

01
Where Lab 04 ends, and what this lab adds
Your file-based pipeline works: checkout, prune, wipe + copy, scan, verify. Nothing is wrong with it. This lab shows the other way teams deploy the same files.
What you have after Lab 04
  • A working deploy pipeline: a merge to main ships files into the running dev gateway; a tag ships prod. Rollback: re-run the pipeline on an older tag.
  • A verified deploy: the pipeline polls the gateway after the scan, so a broken deploy fails the run, loudly.
  • Everything is files in Git: projects, gateway config, and the deploy scripts themselves.
What this lab adds
  • The vocabulary: what an image, a Dockerfile and a container actually are, so "deploy an image" stops being magic.
  • How it's really done: we walk a real production Ignition repo (on screen) that deploys the gateway as a custom image every day: its Dockerfile, its pipeline, its rollback.
  • Your own gateway image: you build it locally, bake a change into it, and deploy it by replacing the dev container. Rollback = run the previous build's tag.
  • The honest trade-off: what image-based buys you, what it costs you, and when you'd pick which. Both approaches stay in your toolbox.
This is a sidestep, not a replacement

Both approaches deploy the same files to the same paths and achieve the same result. File-based mutates a long-lived gateway; image-based replaces a disposable one. Once a pipeline is up and running, you barely notice which one you are on. Today is about understanding the second approach well enough to choose deliberately.

Part 1 · the Dockerfile
02
Read the lab's Dockerfile: a base image plus four COPY lines
This is the actual Dockerfile in this repo. Minimal on purpose: it bakes exactly what Lab 04's pipeline copied at deploy time.
# pin the base: bump deliberately, never float :latest
ARG IGNITION_VERSION=8.3.6
FROM inductiveautomation/ignition:${IGNITION_VERSION}

# bake the deployable state: least to most volatile
COPY third-party-modules/  /third-party-modules/
COPY services/modules.json …/data/modules.json
COPY services/config/      …/data/config/
COPY projects/             …/data/projects/

# provenance: trace any container back to a commit
ARG GIT_SHA=unknown
LABEL org.opencontainers.image.revision="${GIT_SHA}"
What to notice
  • Same paths as Lab 04's docker cp. data/projects and data/config: Ignition scans them at startup. Only when they arrive changed: build time instead of deploy time.
  • FROM = start from the stock gateway. We inherit the whole Ignition install, its entrypoint, EXPOSE 8088 and healthcheck. We only add our files on top.
  • Pinned base version. A floating :latest means two builds a week apart could silently differ. Pin it, bump it on purpose.
  • Per-environment values stay out. Admin password, EULA acceptance, config mode: those are env vars in compose. Config that varies per environment does not belong in a shared artifact.

Part 1 · layers
How layers cache: order the Dockerfile least- to most-volatile

Each COPY is its own layer, cached on its inputs. A busted layer busts every layer after it, so the files you touch daily go last.

FROM inductiveautomation/ignition:8.3.6CACHEDthe base, biggest layer by far
COPY third-party-modules/CACHEDbig .modl binaries · rarely change
COPY services/modules.jsonCACHEDmodule enablement manifest
COPY services/config/CACHEDgateway config
COPY projects/CACHEDproject content · changes daily → last
The rule

Most volatile last. The thing you touch daily should have nothing expensive stacked after it.

See it yourself

Rebuild after touch projects/example-project/project.json — everything still prints CACHED: the cache checksums content, not timestamps. Now append a newline to that file and rebuild: only the projects/ layer rebuilds.

Five layers, bottom-up. Docker reuses any layer whose inputs didn't change.
Part 1 · a production Dockerfile
03
A production Dockerfile: how a real gateway image grows
The lab's Dockerfile is the minimal version. This one comes from a production Ignition repo we maintain (condensed and anonymized): it does more, and every extra line earns its keep. We read the real thing together on screen.
# stage 1: compile a fresh gosu binary (fewer CVEs than the bundled one)
FROM golang:1.25-bookworm AS gosu-builder
# … go build → /out/gosu …

FROM inductiveautomation/ignition:${IGNITION_VERSION}
USER root
COPY --chown=2003:2003 docker_db_migrations/   /docker_db_migrations/
COPY --chown=2003:2003 docker_startup_scripts/ /docker_startup_scripts/
COPY --chown=2003:2003 modules/ignition/ …/data/var/ignition/modl/
COPY --chown=2003:2003 modules/jar/      …/lib/core/gateway/
COPY --chown=2003:2003 pylib/plant       …/user-lib/pylib/plant
COPY --chown=2003:2003 config/           …/data/config
COPY --chown=2003:2003 projects/  /usr/local/share/ignition-seed/projects/

RUN # import company CA certs into the OS + JRE trust stores, \
    # then snapshot data/ into an immutable seed tree: \
    cp -a …/data/. /usr/local/share/ignition-seed/data/

USER 2003:2003
ENTRYPOINT ["/usr/local/bin/entrypoint.sh"]  # then chains to the stock one
What grew, and why
  • More kinds of cargo. Third-party .modl files, plain JAR libraries on the gateway classpath, a Jython library, and a db-migrations folder. All of it versioned, all of it baked. Lab 06 teaches each of these properly.
  • Multi-stage build. Stage 1 exists only to compile one clean binary; the tooling never reaches the shipped image.
  • Non-root runtime. Builds as root, runs as the Ignition UID (2003), hence every COPY --chown.
  • The seed trick. data/ is snapshotted to a seed tree; startup scripts re-seed projects and config from it on every boot. Gateway-UI edits on prod are deliberately erased: the image stays the single source of truth.

Part 1 · the build context
04
.dockerignore: keep the build context small and secret-free
The build context is the folder tarball Docker hands the builder before running the Dockerfile. This is the image-based analogue of Lab 04's .deployignore.
  • Speed. .git/, docs/, exercises/ are megabytes Docker would otherwise upload and checksum on every build. .git is usually the biggest single chunk.
  • Cleanliness. Our Dockerfile only COPYs four paths, but a careless future COPY . . would drag in everything not ignored. Defence in depth.
  • Secrets: the line that matters most. .env is excluded so credentials can never be baked into a shareable artifact. An image gets handed around; a leaked file inside it travels with every copy.
  • Operational state stays out too. .resources/ blob stores and *.idb databases are gateway-owned runtime state (Lab 04's "never commit" list): baking them in would ship one gateway's private state to every other.
# .dockerignore (excerpt)
.git/
.env
.env.example
docs/
exercises/
instructor-notes/
scripts/
# Ignition operational state: never bake
**/.resources/
**/data/db/
**/*.idb
Context ≠ image

The context is what's sent to the builder; the image is what the COPYs select from it. .dockerignore shrinks the former; the Dockerfile selects the latter.

Part 2 · trade-offs
05
File-based vs image-based: advantages and disadvantages
Same files, same target paths, same end result. Each side pays a different price. Know both lists before you pick one for a customer.

File-based (Lab 04)

long-lived gateway
  • +No gateway restart. The container lives on; a hot scan picks up changes in seconds. Runtime state, trial clock and memory tag values stay in place.
  • +Fast inner loop. Copy + scan is a deploy measured in seconds, ideal while actively developing.
  • −Harder to get the file handling right. You must copy the right files to the right places and delete the right ones first. Miss the delete and stale files linger on the gateway forever. Real pipelines get this wrong: we'll look at one that does.
  • −The deploy logic lives in your scripts. Wipe lists, spare lists, .deployignore: all yours to maintain per customer.

Image-based (Lab 05)

disposable gateway
  • +A frozen, versioned artifact. The image you tested is byte-for-byte the image that runs. Rollback and promotion are just running a different tag.
  • +No scan API, no copy scripts. Deploy = replace the container. Feels cleaner: there is nothing to copy and nothing to forget to delete.
  • −Every deploy is a fresh gateway. Internal state is recreated from scratch; anything that must survive (historian data, memory tag values) has to live outside the container, and retaining tag values takes real design work.
  • −Startup downtime. A gateway boot is minutes, not seconds. Either you accept the window or you invest in a solid blue-green setup, and that is real work too.
The honest conclusion

Neither side is free. In the end both achieve the same result, and once the pipeline is up and running you don't really notice which one you are on. What you notice is the failure modes: stale files on one side, lost runtime state and boot windows on the other.

06
The five steps of an image-based deploy
Build, tag, push, pull, run. The image already contains the state, so the deploy itself is just "recreate the container from the new image".
STEP 1

Build

Bake projects / config / modules into layers from the Dockerfile.

→
STEP 2

Tag

An immutable name for this exact build: :sha-<short> or a version like :v0.9.1.

→
STEP 3

Push

Upload to a registry the target machine can reach.

→
STEP 4

Pull

The machine that runs the gateway downloads that tag.

→
STEP 5

Run

Recreate the container from the new image. That is the deploy.

In this lab: steps 1, 2 and 5, all local

A gateway image is around 2 GB. Pushing that through a home connection for every exercise is no fun, so in the assignment you build and tag locally and deploy by recreating the local dev container: same steps 1, 2 and 5, with the registry legs (3 and 4) shown on the next slide instead of exercised. The mechanics are identical; only the transport is missing.

Part 2 · the normal workflow
Where the image is built, stored and deployed when Actions runs it

The same five steps, drawn end to end. Your laptop only ships code; the heavy image traffic happens between GitHub and the registry, never through your connection.

💻

Your laptop

Edit a view, commit on a branch, open a PR, merge to main. A few KB of JSON: that's all that leaves your machine.

→
merge triggers
⚙️

GitHub · Actions

The merge fires deploy.yml, the same trigger as Lab 04. A hosted runner checks out the commit and runs docker build: the Dockerfile bakes the image.

→
docker push
📦

The registry

This is where images live: ghcr.io or a company registry. The ~2 GB upload happens here, datacenter to datacenter, in seconds. Every tag ever pushed stays available.

→
docker pull
🏭

Runner at the plant

The deploy job runs on the machine that owns the gateway: docker pull the new tag, then docker compose up -d recreates the gateway container. That's the deploy.

Part 2 · a real pipeline
07
How a production pipeline builds and deploys the image
The same five steps, as they run in production today in the repo whose Dockerfile you just saw. We walk the real pipeline file on screen; here is the shape to hold on to.
StageWhat happensWorth stealing
VersionA merge to main computes the next version from git history (GitVersion) and pushes a v0.9.1-style tag. The run fails if the tag already exists.Humans never invent version numbers; the pipeline does.
Build + pushThe image is built once and pushed to the company registry, tagged with that version.Build once. Every environment runs a tag of this one build, never a rebuild.
DeployThe "deploy" is docker compose down + up with the new tag: the old container is destroyed, a fresh one boots from the new image. Startup scripts re-seed projects and config from the image on every boot.Deploy and restart are the same operation. There is no drift to manage: the image is the truth.
RollbackPoint the deployed tag back to the previous version and run the same deploy. The old image still exists in the registry, untouched.Rollback is not a special procedure, just yesterday's tag.
Division of labour

The app repo owns the Dockerfile and what goes in the image. The infra side owns where it runs and which env vars it gets (including secrets: that story is Lab 06). Clean seam: the artifact travels, the environment stays put.

Part 2 · state
08
What dies with the container, and what must live outside it
Replacing the container on every deploy is only safe when the data that matters is not in the container. This is the design work the "fresh gateway" disadvantage demands.
Dies with the containerMust live outside
Project content, config, modules (they come back from the image: that's the point)Historian data → TimescaleDB, its own container + volume
The gateway's internal DB, re-commissioned each bootMemory tag values you can't afford to lose → external DB, or don't go image-based for that gateway
Logs, trial clock, anything hand-edited in the gateway UIAudit logs, alarm journal, user uploads → volume or database, chosen deliberately
# db-init/01-create-env-databases.sql
# runs ONCE when the timescaledb volume first initializes
CREATE DATABASE ignition_dev;
CREATE DATABASE ignition_prd;
# (ignition_loc comes from POSTGRES_DB in compose)
Ask this question per data kind

For every piece of state on a gateway: does it belong in the image (comes back on deploy), a volume (survives the container) or a database (survives the machine)? The production repo answers it per folder: user uploads get stashed and restored across the re-seed; everything else is image-owned.
