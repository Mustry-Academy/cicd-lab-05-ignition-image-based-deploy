Bake the gateway into an image.
This morning you shipped files into a running gateway. Now the deployable state (project, config, and modules) becomes an immutable, versioned Docker image. You deploy by replacing the container, promote by re-tagging the exact image you tested, and roll back by running a previous tag.

Teaching
01
Lab 04 left an itch
File-based deploy (docker cp + a hot scan into a running gateway) is a great inner loop. But three things it fundamentally cannot do:
GAP 1

Modules can't ship

A project scan hot-reloads views, scripts and config, but enabling or disabling a module needs a gateway restart. The scan API simply won't apply it.

GAP 2

No versioned artifact

You deployed files. There's nothing to point at, audit, or roll back to: rollback means re-copying old files and re-scanning, with a partial-state window.

GAP 3

Promotion re-copies

dev → prod meant re-shipping files per environment. Nothing guarantees prod received the exact bytes dev tested.

Today

The deployable state becomes an immutable image. And to be clear: this is not a replacement. Most mature Ignition shops use both. File-based for the daily inner loop, image-based for releases. In this lab you'll live in both worlds at once: local stays file-based, dev/prod go image-based.

Block C · mental model
02
Mutate in place vs. replace the box
Same files, same target paths. What changes is when they arrive and what mutates.

File-based (Lab 04)

mutable gateway
  • The gateway is a long-lived, mutable container.
  • Deploy = docker cp files into data/projects & data/config of a running gateway…
  • …then POST the scan endpoints so it picks them up hot.
  • Files arrive at deploy time. The gateway mutates in place.

Image-based (Lab 05)

immutable artifact
  • The deployable state is frozen into image layers at build time.
  • Deploy = docker pull + recreate the container. No copy, no scan API.
  • Files are present from the first boot: Ignition scans them at startup.
  • Nothing mutates. You replace the whole box.
The free lunch

The thing Lab 04 couldn't do, ship a module change, falls out for free. A module change needs a restart, and here the restart is the deploy: "run the new image" and the new module manifest boots with it.

Two deploy philosophies, side by side.
Block C · the Dockerfile
03
The whole Dockerfile is four COPY lines
This is the lab's actual Dockerfile. Everything else is pinning and provenance.
# pin the base: bump deliberately, never float :latest
ARG IGNITION_VERSION=8.3.6
FROM inductiveautomation/ignition:${IGNITION_VERSION}

# bake the deployable state: least to most volatile
COPY third-party-modules/  /third-party-modules/
COPY services/modules.json …/data/modules.json
COPY services/config/      …/data/config/
COPY projects/             …/data/projects/

# provenance: trace any container back to a commit
ARG GIT_SHA=unknown
LABEL org.opencontainers.image.revision="${GIT_SHA}"
What to notice
  • Same paths as Lab 04's docker cp. data/projects and data/config: Ignition scans them at startup. Only when they arrive changed.
  • Pinned base image. An immutable-artifact philosophy is undermined by a floating :latest: two builds a week apart could silently differ.
  • Nothing to override. The IA base image already defines the entrypoint, EXPOSE 8088 and a healthcheck. We inherit all of it.
  • Per-environment knobs stay out. Admin password, EULA, module cert/license acceptance live as env vars in compose: config that varies per env doesn't belong in a shared artifact.

Block C · layers
Layer order is a cache strategy

Each COPY is its own layer, cached on its inputs. A busted layer busts every layer after it, so order from least- to most-frequently-changed.

FROM inductiveautomation/ignition:8.3.6CACHEDthe base, biggest layer by far
COPY third-party-modules/CACHEDbig .modl binaries · rarely change
COPY services/modules.jsonCACHEDmodule enablement manifest
COPY services/config/CACHEDgateway config
COPY projects/CACHEDproject content · changes daily → last
The rule

Most volatile last. The thing you touch daily should have nothing expensive stacked after it.

See it yourself

touch projects/example-project/project.json then rebuild, and watch which layers print CACHED in the build output.

Five layers, bottom-up. Docker reuses any layer whose inputs didn't change.
Block C · modules
04
Modules ride inside the image
Two pieces bake together: the binaries and the enablement manifest. This is the Lab 04 payoff.
third-party-modules/ · the binaries
  • Bundled .modl files: Embr Charts, Embr Periscope, MQTT Engine, TimescaleDB Historian.
  • Loaded via -Dignition.gateway.externalModulesFolder=/third-party-modules (see the compose command:).
  • Baked as the first COPY layer: large and rarely changing, so it's almost always CACHED.
  • External JAR libraries follow the same move. In 8.3 even JDBC drivers ship as modules: modules.json enables com.inductiveautomation.jdbc.postgresql right here. Anything else JAR-shaped is just one more COPY layer: binaries ship with the artifact, not by hand.
# services/modules.json: which modules turn on at boot
"com.cirruslink.mqtt.engine.gateway": {
  "filename": "/third-party-modules/MQTT-Engine-signed.modl",
  "onStartup": "enabled",
  "certFingerprint": "b3d505…",
  "licenseAgreementHash": 1007003261
}
Why this can't scan

modules.json is read at boot. A scan can't enable a module: only a restart can. When the manifest is a layer in the image, "restart with the new module" is just "run the new image." The cert fingerprint + license hash let the gateway accept it headlessly.

Block C · the build context
05
.dockerignore · guard the build context
The build context is the tarball Docker hands the daemon before running the Dockerfile. This is the image-based analogue of Lab 04's .deployignore, moved into the tool.
  • Speed. .git/, docs/, exercises/ are megabytes Docker would otherwise upload and checksum on every build. .git is usually the biggest single chunk.
  • Cleanliness. Our Dockerfile only COPYs four paths, but a careless future COPY . . would drag in everything not ignored. Defence in depth.
  • Secrets: the line that matters most. .env is excluded so credentials can never be baked into a published, pullable artifact.
  • It's enforced. scripts/validate.sh and ci.yml both assert the must-exclude patterns are present: .env, .git/, scripts/, docs/, exercises/.
# .dockerignore (excerpt)
.git/
.env
.env.example
docs/
exercises/
instructor-notes/
scripts/
# Ignition operational state: never bake
**/.resources/
**/data/db/
**/*.idb
Context ≠ image

The context is what's sent to the daemon; the image is what the COPYs select from it. .dockerignore shrinks the former; the Dockerfile selects the latter.

Block C · naming builds
06
One image, several names
Deploy immutable tags, navigate with moving tags. And stamp every image with the commit it came from.
:sha-<short>immutableThe thing you actually deploy. Always points at exactly one build.
:devmoving"Whatever's latest on dev." A convenient pointer, but never deploy by this name; it moves under you.
:vX.Y.ZimmutableA released version. A re-tag of the tested image, no rebuild.
:prodmoving"Whatever's live in prod." Same caveat as :dev.
:localmovingYour latest local build: what scripts/build-image.sh tags alongside :sha-….
# CI passes the commit into the build…
docker build --build-arg GIT_SHA=$(git rev-parse --short HEAD) …

# …and months later you can always ask a running
# container which commit it is:
docker inspect -f \
  '{{ index .Config.Labels "org.opencontainers.image.revision" }}' \
  cicd-lab-05-ignition:local
The question this answers

"What commit is prod actually running?" With the revision label, that's one docker inspect away, not an archaeology project.

Block D · the pattern
07
The pattern in five steps
No SSH-ing files, no scan API. The image already contains the state, so you replace the whole container, then verify health.
STEP 1

Build

Bake projects / config / modules into layers from the Dockerfile.

→
STEP 2

Tag

:sha-<short> (immutable) + a moving pointer (:dev).

→
STEP 3

Push

To the registry: GHCR, under your fork's namespace.

→
STEP 4

Pull

On the runner that owns the target gateway.

→
STEP 5

Run

docker compose up -d ignition-<env> recreates the container.

Contrast with Lab 04

File-based was checkout → prune → ship → scan → verify: the artifact was files and the gateway stayed alive. Here the artifact is an image and the gateway is replaced. Old container runs until the new one is ready, then swaps, so the rm -rf + docker cp partial-state window is gone entirely.

Block D · topology
08
Build is portable. Deploy is not.
In Lab 04 the self-hosted runner did everything. Here it only does the last mile, and that split is the operational payoff.

GitHub Actions · ubuntu-latest

The build job: docker build + push. Just needs Docker + registry creds, nothing environment-specific. Free hosted minutes, off your privileged runner.

→
build + push

GHCR · your namespace

ghcr.io/<you>/cicd-lab-05-ignition
holds :sha-…, :dev, :vX.Y.Z, :prod. Auth = the workflow's built-in GITHUB_TOKEN.

→
docker pull

Self-hosted runner · [self-hosted, lab05]

The deploy job: pull + docker compose up -d. Must reach the gateway container's Docker daemon, which is why it can't be hosted.

Two footnotes that bite

1. .github/actionlint.yaml declares the custom lab05 label so actionlint doesn't flag it as unknown. 2. The runner mounts the Docker socket: lab-grade, not production-grade. Real setups use a pull-based agent on the gateway host or an orchestrator (that discussion is the Block D debrief).

Block D · Git Flow

Git Flow: two branches, two gateways
Every long-lived branch maps to a deployed environment. Three workflows do all the work.

The local gateway stays file-based

Environments = gates

Each deploy job runs in a GitHub environment (lab-gateway-dev / lab-gateway-prod): per-stage deploy history for free, and the place to bolt required reviewers onto prod, a manual approval gate with zero workflow changes.

Block D · the centerpiece
10
Promote, don't rebuild
The single most important idea of the day: on a release tag, nothing is rebuilt. The tested image is re-tagged, server-side.
# release.yml on tag v0.1.0: a manifest copy, no layers move
docker buildx imagetools create \
  --tag <repo>:v0.1.0 --tag <repo>:prod \
  <repo>:dev     # the exact image dev validated

# then prove it: same digest on both gateways:
docker inspect -f '{{ index .RepoDigests 0 }}' lab05-ignition-dev
docker inspect -f '{{ index .RepoDigests 0 }}' lab05-ignition-prod
# identical sha256:… means prod runs the bytes dev tested
  • Why :dev and not the tagged commit? A Git Flow release is a merge into main, a new commit SHA no image was ever built for. So you promote what dev is running, not "by commit."
  • Why not just rebuild from the tag? A rebuild could pull a newer base layer or a non-deterministic dependency, so you'd ship bytes dev never saw. Build-once/promote-many makes that impossible by construction.
  • The release/* branch is your freeze point. Cut it when develop (and therefore the :dev image) is exactly what you want in prod. (In the lab, a plain merge stands in for it.)
The money moment

The digest-equality proof is Part 4 of the You-do. If the digests could differ, what would that mean, and why would it scare you?

Block D · rollback
11
Rollback = run the previous tag
Three patterns, increasing maturity. Image-based makes the strong ones cheap.
PATTERN 1

git revert + re-merge

Triggers a fresh build/deploy of the reverted state. Works, but it rebuilds, and you wait for the whole pipeline.

PATTERN 2 · THE CANONICAL ONE

Re-promote a known-good tag

release.yml's workflow_dispatch takes a version input: point prod at v0.1.0's existing image in two clicks. No rebuild, no git surgery, zero irreversible steps.

PATTERN 3

Keep N versions warm

Every release is an immutable tag in the registry, so rollback is always available for any version you haven't deleted. The real question becomes tag retention.

Compare to Lab 04

File-based rollback meant re-copying old files and re-scanning: more steps, partial-state risk, nothing immutable to point at. Ask of each path: how many irreversible steps? Image-based: zero, because you're pointing at an artifact that already exists.

Block D · state
12
What lives where, and what's safe to throw away
Recreating the container on every deploy is only safe because the durable data isn't in the container.
In the image: dies each deployOutside: persists
Project content, config, module manifest + binariesHistorian data → TimescaleDB
The gateway's commissioned internal DB (re-created each boot)Anything you deliberately put on an external volume or DB
The real-world decision

dev/prod here have no persistent volume, intentionally. In a real deployment, gateway audit logs, alarm journal, named user accounts each force the question: does this belong in the image, a volume, or a database?

# db-init/01-create-env-databases.sql
# runs ONCE when the timescaledb volume first initializes
CREATE DATABASE ignition_dev;
CREATE DATABASE ignition_prd;
# (ignition_loc comes from POSTGRES_DB in compose)
The compose stack: one DB, three databases
  • One TimescaleDB, three logical databases: ignition_loc / _dev / _prd, and each gateway points at its own.
  • Each gateway passes -Dignition.config.mode=loc|dev|prd so config can vary per environment.
  • Historian data lives here, which is exactly why throwing away the dev/prod container on each deploy is safe.

Trade-offs
13
When to pick which
Lab 04 planted this table as a preview, and you've now run both sides, so let's settle it. Not a versus but a division of labour: file-based for the inner loop, image-based for releases.
ConcernFile-based (Lab 04)Image-based (Lab 05)
Time to deploySeconds: cp + scanMinutes: build → push → pull → boot
Hot reloadYes: a scan picks up project edits liveNo: every deploy is a container restart
Module changesCan't: a scan won't apply themTrivial: baked in
Gateway impactMutates a live gateway: rm -rf + cp leaves a partial-state windowReplaced whole: old container serves until the new one is up; durable state must live outside (TimescaleDB)
RollbackHard: re-copy old filesEasy: run the previous tag
Promotion across envsRe-copy per envRe-tag the tested image
Auditable artifactNone: just gitA versioned, labelled image with a digest
Best forActive development, frequent project editsReleases, module/baseline changes, immutable infra
When NOT to go image-based

Sub-minute iteration on project content (rebuild + reboot per view tweak is pain, so use the file-based loop) · stateful gateway data you can't externalize (fix the state location first) · no registry reachable from the target (air-gapped sites need docker save/load or a local registry mirror). Full write-up: docs/image-based-deploy-pattern.md.

Block D · registry auth
14
GHCR auth: least privilege, zero secrets
No registry account, no API key, no PAT for the registry. The workflow's built-in token does everything.
  • Push: the build/promote jobs request permissions: packages: write and log in with the job-scoped GITHUB_TOKEN. It expires with the run.
  • Pull: the deploy jobs log in with the same token. Pulling only needs packages: read, which write includes.
  • Your namespace, always. The workflows derive ghcr.io/<owner>/… from github.repository_owner, so a student's fork publishes under their account, never Mustry's.
  • First push creates the package, private by default, and that's fine: the runner authenticates.
  • Don't hand the runner a broad PAT "to be safe." The job token is enough: that's the point.
# deploy.yml (excerpt)
permissions:
  contents: read
  packages: write

- uses: docker/login-action@v3
  with:
    registry: ghcr.io
    username: ${{ github.actor }}
    password: ${{ secrets.GITHUB_TOKEN }}
The classic 403

Fork under an org? Settings → Actions → General → Workflow permissions must be read and write, and org Packages enabled. Personal forks work out of the box. Full list in docs/TROUBLESHOOTING.md.
